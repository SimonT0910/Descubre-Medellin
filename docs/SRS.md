# SRS - Software Requirements Specification
## Descubre Medellín — Blog interactivo de lugares del Valle de Aburrá

---

## Requerimientos Funcionales

Los siguientes requerimientos funcionales describen las capacidades que el sistema debe ofrecer a sus usuarios. Cada requerimiento incluye un identificador único, descripción, prioridad y criterios de aceptación verificables.

| ID | Requerimiento Funcional | Prioridad | Criterios de Aceptación |
|----|------------------------|-----------|--------------------------|
| RF01 | El sistema debe permitir a cualquier usuario consultar el listado general de lugares publicados. | Alta | Dado que el usuario abre la vista principal (Home o Lista de lugares), cuando el frontend realiza `GET /lugares`, entonces se renderizan tarjetas con nombre, categoría, imagen y precio de cada lugar. Si no hay lugares, se muestra un estado vacío con un mensaje informativo. |
| RF02 | El sistema debe permitir registrar una cuenta de usuario con nombre, correo electrónico y contraseña. | Alta | Dado un formulario de registro válido, cuando el usuario envía los datos mediante `POST /auth/register`, entonces el backend crea el usuario en la base de datos y responde con `201 Created`. Si el correo ya existe, el backend responde con `400` y el frontend muestra el mensaje "Este correo ya está registrado". |
| RF03 | El sistema debe permitir iniciar sesión con correo y contraseña. | Alta | Dado el formulario de login con credenciales válidas, cuando se realiza `POST /auth/login`, entonces el servidor responde `200 OK` con un token o sesión activa y el frontend redirige al usuario al inicio. Si las credenciales son incorrectas, el servidor responde `401` y el frontend muestra "Credenciales inválidas". |
| RF04 | El sistema debe permitir a un usuario autenticado publicar un nuevo lugar con nombre, descripción, categoría, precio y al menos una imagen. | Alta | Dado un formulario de publicación con todos los campos requeridos y un usuario autenticado, cuando se envía `POST /lugares`, entonces el backend valida los datos, guarda el lugar en la base de datos y responde `201 Created`. El lugar publicado aparece inmediatamente en el listado general. Si faltan campos obligatorios, el backend responde `400` y el frontend muestra los errores de validación correspondientes. |
| RF05 | El sistema debe permitir ver el detalle completo de un lugar, incluyendo nombre, descripción, categoría, precio, imágenes, calificación promedio y comentarios. | Alta | Dado que el usuario selecciona un lugar desde el listado, cuando el frontend realiza `GET /lugares/:id`, entonces se muestra la vista de detalle con toda la información del lugar y la lista de comentarios obtenida de `GET /lugares/:id/comentarios`. Si el lugar no existe, el backend responde `404` y el frontend muestra un mensaje de error. |
| RF06 | El sistema debe permitir buscar lugares por nombre o descripción. | Alta | Dado el campo de búsqueda en la vista de listado, cuando el usuario escribe un término y el frontend consulta `GET /lugares` con el parámetro de búsqueda, entonces se muestran únicamente los lugares cuyo nombre o descripción contengan el término. Si no hay resultados, se muestra un estado vacío con el mensaje "No se encontraron lugares para tu búsqueda". |
| RF07 | El sistema debe permitir filtrar lugares por categoría y por rango de precio. | Alta | Dado que el usuario selecciona una categoría o define un rango de precio en los filtros disponibles, cuando el frontend consulta `GET /lugares?categoria=X` o `GET /lugares?precioMin=Y&precioMax=Z`, entonces se muestran solo los lugares que cumplan con el filtro aplicado. Los filtros pueden combinarse. |
| RF08 | El sistema debe permitir a un usuario autenticado agregar un lugar a su lista de favoritos. | Media | Dado que el usuario está autenticado y visualiza el detalle o la tarjeta de un lugar, cuando presiona "Agregar a favoritos" y el frontend envía `POST /lugares/:id/favoritos` con el token de autenticación, entonces el backend valida que el lugar no esté ya en favoritos del usuario, lo guarda en la tabla Favoritos y responde `201 Created`. El ícono de favorito se actualiza visualmente. Si el lugar ya estaba guardado, el backend responde con un mensaje indicándolo y no crea un duplicado. |
| RF09 | El sistema debe permitir a un usuario autenticado consultar su lista de lugares favoritos. | Media | Dado que el usuario autenticado navega a su perfil o a la vista de favoritos, cuando el frontend realiza `GET /lugares/:id/favoritos` (o el endpoint equivalente de favoritos del usuario), entonces se muestran todas las tarjetas de los lugares guardados. Si no tiene favoritos, se muestra un estado vacío con el mensaje "Aún no tienes lugares favoritos". |
| RF10 | El sistema debe permitir a un usuario autenticado comentar en el detalle de un lugar. | Media | Dado que el usuario autenticado está en la vista de detalle de un lugar y escribe un comentario no vacío, cuando envía el formulario y el frontend realiza `POST /lugares/:id/comentarios`, entonces el backend guarda el comentario asociado al usuario y al lugar, responde `201 Created` y el nuevo comentario aparece al instante en la lista sin recargar la página. Si el comentario está vacío, el frontend muestra la validación "El comentario no puede estar vacío". |
| RF11 | El sistema debe mostrar la calificación promedio de cada lugar calculada a partir de las valoraciones de los usuarios. | Media | Dado que un lugar tiene calificaciones registradas en la base de datos, cuando el frontend carga el detalle del lugar, entonces se muestra el promedio calculado por el backend (p. ej. 4.2 / 5). Si el lugar no tiene calificaciones, se muestra "Sin calificaciones aún". El promedio se recalcula automáticamente cuando se añade una nueva valoración. |
| RF12 | El sistema debe permitir al usuario cerrar sesión desde cualquier pantalla. | Alta | Dado que el usuario tiene una sesión activa, cuando presiona el botón "Cerrar sesión", entonces el frontend elimina el token o sesión almacenada, redirige al usuario al inicio y las opciones que requieren autenticación (publicar, comentar, favoritos) dejan de estar disponibles. |

---

> **Nota:** Los requerimientos RF01, RF05, RF06 y RF07 están disponibles para usuarios no registrados (visitantes). Los requerimientos RF04, RF08, RF09, RF10 y RF12 requieren autenticación activa. RF02 y RF03 aplican exclusivamente a usuarios no autenticados.
